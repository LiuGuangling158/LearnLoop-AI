## 1. 项目背景与痛点

### 1.1 痛点场景

| 痛点 | 描述 | 本系统解决方案 |
|------|------|---------------|
|  笔记碎片化 | 学习资料散落在 Markdown、PDF、课件各处，难以统一检索 | RAG 知识库统一索引，自然语言检索 |
|  缺乏自测手段 | 学完不知道掌握程度，缺少针对性的练习题 | Quiz Agent 自动生成多题型、多难度题目 |
|  错题无法追踪 | 做错的题没有系统记录，同类错误反复出现 | Memory Agent 构建错题本 + 遗忘曲线追踪 |
|  复习无计划 | 不知道今天该复习什么，遗忘曲线无法利用 | Learning Scheduler 基于 SM-2 算法自动规划 |
|  多模型切换难 | 不同任务适合不同模型，手动切换成本高 | LLM Layer 统一抽象，按任务路由到最优模型 |

### 1.2 目标用户画像

- **软件测试/QA 学习者**：需要系统化掌握 ISTQB、测试方法论
- **技术面试备考者**：需要大量刷题 + 错题追踪
- **自学者群体**：有学习资料但缺乏结构化和自测闭环

### 1.3 项目愿景

> 构建一个 **"学-练-测-记-复"五步闭环** 的 AI 学习助手，通过 Multi-Agent 协作实现个性化学习体验，并作为展示 AI Agent 工程能力的作品集项目。

## 2. 核心概念与技术差异化

### 2.1 这不是一个单 Agent 应用

```
传统 AI 学习工具：
  User → Chatbot → LLM → 回复
  
本系统：
  User → API Gateway → Agent Orchestrator → 多个专业 Agent 协作
                                              ├── 每个 Agent 有独立 Prompt Template
                                              ├── Agent 间通过 Memory 共享上下文
                                              ├── Orchestrator 做任务分解和路由
                                              └── 支持多模型按任务分派
```

### 2.2 核心技术特征

| 特征 | 实现方式 | 面试加分点 |
|------|---------|-----------|
| **Multi-Agent Orchestration** | 自研 Orchestrator 负责 Agent 注册、路由、协作 | 体现对 Agent 架构的深度理解 |
| **Tool-Calling** | 每个 Agent 可调用 VectorDB、文件系统、外部 API | 展示 Function Calling 实战能力 |
| **Memory System** | 短期对话记忆 + 长期知识记忆（VectorDB） | 体现对记忆机制的工程化思考 |
| **RAG Pipeline** | 文档 Chunking → Embedding → 向量检索 → Reranking | 展示 RAG 全链路工程能力 |
| **LLM-as-Judge** | 批改 Agent 使用 Rubric 评分体系 | 体现对 LLM 评估范式的理解 |
| **Structured Output** | 所有 Agent 输出均为结构化 JSON → 前端渲染 | 体现 Prompt Engineering 深度 |

## 4. 核心功能模块详解

### 4.1 Note Agent（学习整理）

**触发词：** "帮我整理XX知识" / "生成XX笔记"

**输入格式：**
```json
{
  "topic": "软件测试方法",
  "source_material": ["文件路径 或 粘贴文本"],
  "style": "detailed | summary | mindmap",
  "language": "zh-CN"
}
```

**内部流程：**
```
原始输入 → Prompt Chaining (3 阶段)
  阶段1: 知识提取 → 识别关键概念、定义、分类
  阶段2: 结构化整理 → 按层级/类型/关系组织
  阶段3: Markdown 生成 → 含目录/表格/高亮/代码块
→ 结构化输出 (JSON Schema 校验) → 存入 VectorDB
```

**输出格式：**
```json
{
  "title": "软件测试方法概述",
  "sections": [
    {
      "heading": "测试类型分类",
      "level": 2,
      "content": "...",
      "subsections": [...],
      "key_points": ["单元测试", "集成测试", "系统测试"]
    }
  ],
  "summary": "本文涵盖了软件测试的三大分类维度...",
  "tags": ["测试", "QA", "ISTQB"],
  "embedding_id": "vec_note_20240623_001"
}
```

**技术亮点：**
- Prompt Chaining：将"提取→整理→生成"拆分为 3 个 Prompt，每步可独立调优
- Structured Output：使用 JSON Schema 约束输出格式，前端直接渲染
- 自动标签 + 摘要：后续 RAG 检索的基础

### 4.2 Quiz Agent（出题系统）

**触发词：** "出题" / "来几道题" / "测验一下"

**输入格式：**
```json
{
  "topic": "软件测试基础",
  "question_types": ["choice", "short_answer", "dictation", "true_false"],
  "difficulty": "easy | medium | hard",
  "count": 10,
  "mode": "standard | reverse",
  "source_notes": ["vec_note_id_1", "vec_note_id_2"]
}
```

**内部流程：**
```
指定主题/源笔记 → RAG 检索相关知识块
  → Prompt Template 注入: 知识点 + 题型 + 难度 + 数量
  → LLM 批量生成题目
  → Schema 校验（每题必须有: id/type/question/options/answer/explanation）
  → 存入题库 DB
  → 返回前端渲染
```

**支持的题型与输出 Schema：**
```json
{
  "quiz_id": "quiz_20240623_001",
  "questions": [
    {
      "id": "q_001",
      "type": "choice",
      "difficulty": "medium",
      "question": "以下哪项属于黑盒测试方法？",
      "options": [
        {"key": "A", "text": "等价类划分"},
        {"key": "B", "text": "语句覆盖"},
        {"key": "C", "text": "条件覆盖"},
        {"key": "D", "text": "路径覆盖"}
      ],
      "answer": "A",
      "explanation": "等价类划分关注输入/输出，不关心内部逻辑，属于黑盒测试方法。B/C/D 均为白盒测试方法。",
      "source_note": "软件测试方法 - 黑盒测试部分"
    },
    {
      "id": "q_002",
      "type": "dictation",
      "mode": "reverse",
      "prompt": "故障（中文→英文翻译）",
      "answer": "Fault",
      "explanation": "Fault 是缺陷的静态表现，区别于 Failure（运行时表现）"
    }
  ],
  "metadata": {
    "total": 10,
    "difficulty_distribution": {"easy": 3, "medium": 5, "hard": 2},
    "generated_by": "deepseek-chat"
  }
}
```

**技术亮点：**
- Prompt Template System：每种题型有独立模板，组合使用
- Difficulty Calibration：通过 Prompt 控制 Bloom 认知层级（记忆→理解→应用→分析）
- Reverse Quiz：支持中→英 / 英→中双向默写，适合术语记忆

### 4.3 Grading Agent（自动批改）

**触发词：** 用户提交答案后自动触发

**输入格式：**
```json
{
  "quiz_id": "quiz_20240623_001",
  "submissions": [
    {
      "question_id": "q_001",
      "user_answer": "A"
    },
    {
      "question_id": "q_002",
      "user_answer": "Fault"
    }
  ]
}
```

**内部流程：**
```
用户答案 + 标准答案
  → Rubric 评分引擎
    ├── 客观题：精确匹配 → 0/1 计分
    ├── 简答题：Semantic Matching → 0-100 分
    │   ├── 关键词覆盖度 (40%)
    │   ├── 语义相似度 (30%)
    │   └── 逻辑完整性 (30%)
    └── 英文默写题：字符级相似度 + 词义匹配
  → 错题分析
    ├── 定位错误知识点
    ├── 生成纠错建议
    └── 推荐强化练习
  → 提交到 Memory Agent
```

**输出格式：**
```json
{
  "grading_id": "grade_20240623_001",
  "total_score": 85,
  "results": [
    {
      "question_id": "q_001",
      "is_correct": true,
      "score": 100,
      "user_answer": "A",
      "correct_answer": "A"
    },
    {
      "question_id": "q_002",
      "is_correct": false,
      "score": 0,
      "user_answer": "Fault",
      "correct_answer": "Fault",
      "feedback": "拼写正确！但你之前将 Fault 和 Failure 混淆，建议回顾两者区别。",
      "reinforcement_suggestion": "创建对比卡片: Fault vs Failure vs Error"
    }
  ],
  "error_analysis": {
    "weak_points": ["黑盒测试 vs 白盒测试区分"],
    "error_pattern": "概念混淆型错误",
    "recommended_review": ["note_vec_003"]
  }
}
```

**技术亮点：**
- LLM-as-Judge：利用 LLM 的语义理解能力判断简答题
- Rubric Scoring：多维度评分，而非简单的对/错
- 错误模式识别：不只是"你错了"，而是"你哪类问题经常错"

### 4.4 Memory Agent（错题本系统）

**触发方式：** 自动（Grading Agent 完成后触发）/ 手动（用户查询错题）

**存储结构：**
```
短时记忆 (Conversation Buffer)
  └── 当前会话的上下文窗口 (最近 N 轮对话)

长时记忆 (VectorDB + SQL)
  ├── 知识点 Embedding
  ├── 错题记录 (关联知识点)
  ├── 易混概念对 (Fault↔Failure, Verification↔Validation)
  └── 错误频率统计
```

**核心功能：**

| 功能 | 描述 | 实现 |
|------|------|------|
| 错题录入 | 自动从 Grading Agent 获取错题 | Event-driven |
| 薄弱点分析 | 统计错误频次 → 生成弱点报告 | SQL 聚合 + LLM 总结 |
| 相似题推荐 | 根据错题知识点检索相似题目 | Vector Similarity Search |
| 易混点检测 | 检测用户反复混淆的概念对 | Pattern Matching |
| 遗忘提醒 | 错题到 N 天后推送重做 | SM-2 算法联动 |

**输出示例 (弱点报告)：**
```json
{
  "user_id": "u_001",
  "report_period": "2024-06-01 ~ 2024-06-23",
  "error_statistics": {
    "total_errors": 23,
    "by_topic": {
      "黑盒测试": 8,
      "测试用例设计": 6,
      "缺陷管理": 5,
      "自动化测试": 4
    },
    "by_type": {
      "概念混淆": 12,
      "知识遗忘": 7,
      "理解偏差": 4
    }
  },
  "top_confusion_pairs": [
    {"concept_a": "Verification", "concept_b": "Validation", "error_count": 5},
    {"concept_a": "Fault", "concept_b": "Failure", "error_count": 3}
  ],
  "improvement_suggestions": [
    "建议重点复习 Verification vs Validation 的区别",
    "Fault/Error/Failure 三角关系建议制作对比表格"
  ]
}
```

**技术亮点：**
- 双存储架构：VectorDB（语义相似）+ SQL（统计聚合），各取所长
- 易混概念自动发现：检测用户重复犯错的概念对
- 与 Scheduler 联动：错题数据驱动复习计划

### 4.5 Retrieval Agent（RAG 知识检索）🔍

**触发词：** "什么是XX" / "XX和XX的区别"

**内部流程：**
```
用户 Query
  → Query Rewriting (扩展/纠错/拆解)
  → Embedding 生成
  → VectorDB 检索 (Top-K, K=10)
  → Reranking (Cross-Encoder 精排)
  → Context 拼接
  → LLM 生成 (带引用)
  → 返回答案 + 来源引用
```

**Chunking Strategy（分块策略）：**
```
Markdown 文档
  → 按标题层级切分 (H1/H2/H3 为边界)
  → 每块 500-1000 tokens
  → 保留 100 token 重叠 (overlap)
  → 保留元数据 (来源文件、标题路径、页码)
```

**Reranking 策略：**
```
VectorDB 粗排 (Bi-Encoder, Top-50)
  → Cross-Encoder 精排 (Top-10)
  → 去重过滤 (相似度 > 0.9 的去重)
  → 返回最终 Top-K
```

**输出格式：**
```json
{
  "query": "Verification 和 Validation 的区别是什么？",
  "answer": "Verification（验证）是确保产品符合规格说明——'我们正确地建造产品了吗？'，关注过程。Validation（确认）是确保产品满足用户需求——'我们建造了正确的产品吗？'，关注结果。",
  "sources": [
    {
      "note_title": "软件测试基础 - ISTQB第一章",
      "chunk_id": "chunk_042",
      "relevance_score": 0.92,
      "excerpt": "Verification: Are we building the product right?..."
    }
  ],
  "related_concepts": ["软件测试", "ISTQB", "质量保证"]
}
```

**技术亮点：**
- Query Rewriting：用户的问题可能不精确，先用 LLM 改写为更好的检索 Query
- Two-Stage Retrieval：Bi-Encoder 粗排 + Cross-Encoder 精排
- 来源引用可追溯：每个回答都能溯源到原始笔记

### 4.6 Learning Scheduler（学习规划器）📅

**触发方式：** 定时触发 / 用户打开 Dashboard 时

**核心算法：SM-2 (SuperMemo 2)**
```
EF (Easiness Factor) 初始值 = 2.5
Interval (复习间隔) 初始值 = 1 天

评分 (0-5):
  0-2: 完全忘记 → Interval 重置为 1, EF 降低 0.2
  3:   勉强记住 → Interval 重置为 1, EF 不变
  4:   正确回忆 → Interval = Interval × EF
  5:   非常轻松 → Interval = Interval × EF, EF 增加 0.1

下次复习日期 = 今天 + Interval 天
```

**每日任务生成流程：**
```
1. 从 Memory Agent 获取所有知识点
2. 筛选：due_date <= today 的待复习项
3. 按优先级排序:
   ├── 高优先级: overdue + 错误次数 > 3
   ├── 中优先级: due today + 错误次数 1-3
   └── 低优先级: due today + 错误次数 0
4. 组装学习任务卡片
5. 返回前端 Dashboard
```

**输出格式 (每日任务)：**
```json
{
  "date": "2024-06-23",
  "daily_tasks": [
    {
      "type": "review",
      "knowledge_point": "黑盒测试方法",
      "reason": "上次错误，距今天 3 天，需复习",
      "priority": "high",
      "suggested_duration_min": 15,
      "linked_quiz_id": "quiz_review_blackbox_001"
    },
    {
      "type": "new_learn",
      "knowledge_point": "性能测试基础",
      "reason": "学习计划中的新内容",
      "priority": "medium",
      "suggested_duration_min": 30
    }
  ],
  "total_estimated_time_min": 45,
  "streak_days": 7
}
```

## 5. API 接口设计

### 5.1 总体约定


## 6. 数据模型设计

### 6.1 关系型数据库 (SQLite / PostgreSQL)

```sql
-- 用户表
CREATE TABLE users (
    id          TEXT PRIMARY KEY,
    username    TEXT UNIQUE NOT NULL,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 笔记表
CREATE TABLE notes (
    id              TEXT PRIMARY KEY,
    user_id         TEXT REFERENCES users(id),
    title           TEXT NOT NULL,
    content_md      TEXT NOT NULL,         -- Markdown 原文
    summary         TEXT,                   -- AI 生成的摘要
    tags            TEXT,                   -- JSON Array: ["测试","ISTQB"]
    source_type     TEXT,                   -- 'generated' | 'uploaded' | 'manual'
    embedding_id    TEXT,                   -- VectorDB 中的对应 ID
    word_count      INTEGER,
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 题目表
CREATE TABLE quizzes (
    id              TEXT PRIMARY KEY,
    user_id         TEXT REFERENCES users(id),
    topic           TEXT NOT NULL,
    difficulty      TEXT CHECK(difficulty IN ('easy','medium','hard')),
    questions_json  TEXT NOT NULL,          -- JSON Array of question objects
    source_note_ids TEXT,                   -- JSON Array
    generated_by    TEXT,                   -- 使用的 LLM 模型
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 做题记录表
CREATE TABLE quiz_attempts (
    id              TEXT PRIMARY KEY,
    user_id         TEXT REFERENCES users(id),
    quiz_id         TEXT REFERENCES quizzes(id),
    answers_json    TEXT NOT NULL,          -- 用户提交的答案
    score           REAL,
    graded_by       TEXT,                   -- 批改 Agent ID
    grading_json    TEXT,                   -- 完整批改结果
    completed_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 错题记录表
CREATE TABLE error_log (
    id              TEXT PRIMARY KEY,
    user_id         TEXT REFERENCES users(id),
    question_id     TEXT NOT NULL,
    quiz_id         TEXT REFERENCES quizzes(id),
    attempt_id      TEXT REFERENCES quiz_attempts(id),
    user_answer     TEXT,
    correct_answer  TEXT,
    error_type      TEXT,                   -- '概念混淆' | '知识遗忘' | '理解偏差'
    knowledge_point TEXT,                   -- 关联的知识点
    reviewed_count  INTEGER DEFAULT 0,      -- 已复习次数
    is_resolved     BOOLEAN DEFAULT FALSE,  -- 是否已掌握
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 学习计划表
CREATE TABLE learning_plans (
    id              TEXT PRIMARY KEY,
    user_id         TEXT REFERENCES users(id),
    topic           TEXT NOT NULL,
    goal_description TEXT,
    target_date     DATE,
    status          TEXT CHECK(status IN ('active','paused','completed')),
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- SM-2 记忆状态表
CREATE TABLE sm2_states (
    id              TEXT PRIMARY KEY,
    user_id         TEXT REFERENCES users(id),
    knowledge_point TEXT NOT NULL,
    ef              REAL DEFAULT 2.5,       -- Easiness Factor
    interval_days   INTEGER DEFAULT 1,      -- 当前间隔
    repetitions     INTEGER DEFAULT 0,      -- 重复次数
    next_review_at  DATE,                   -- 下次复习日期
    last_score      INTEGER,                -- 上次评分 0-5
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 易混概念对表
CREATE TABLE confusion_pairs (
    id              TEXT PRIMARY KEY,
    user_id         TEXT REFERENCES users(id),
    concept_a       TEXT NOT NULL,
    concept_b       TEXT NOT NULL,
    error_count     INTEGER DEFAULT 1,
    last_confused_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 8. 技术栈详解

### 8.1 后端核心

| 组件 | 候选方案 | 推荐理由 |
|------|---------|---------|
| **Web 框架** | FastAPI | 异步原生支持、自动 OpenAPI 文档、SSE 流式友好 |
| **Agent 框架** | LangChain / LlamaIndex / 自研 | LangChain 生态全但抽象层多；自研更灵活且能展示深度 |
| **向量数据库** | ChromaDB (MVP) / FAISS (进阶) | ChromaDB 轻量零配置适合 MVP；FAISS 性能更强适合大规模 |
| **关系数据库** | SQLite (MVP) / PostgreSQL (进阶) | SQLite 零配置快速起步；PostgreSQL 支持向量插件 pgvector |
| **缓存/消息** | Redis (可选) | Agent 间任务队列、会话缓存 |

## 10. 面试亮点包装

### 10.1 项目一句话介绍（电梯演讲）

> "我开发了一个 AI 驱动的个性化学习系统，核心是 **Multi-Agent 协作架构**——包含笔记生成、自动出题、智能批改、错题追踪和 RAG 知识检索 5 个专业 Agent，通过自研的 Orchestrator 进行任务编排。技术栈是 Python/FastAPI + ChromaDB + 多 LLM 抽象层，实现了完整的'学-练-测-记-复'学习闭环。"

### 10.2 高频追问 & 回答要点

| 面试官可能问 | 回答要点 |
|-------------|---------|
| **"为什么不用单 Agent？"** | 单 Agent 难以处理复杂多步骤任务；Multi-Agent 分工明确，每个 Agent 的 Prompt 和策略可独立优化；Agent 间通过 Memory 共享上下文而非全部塞进一个 Prompt |
| **"Orchestrator 怎么实现的？"** | 意图识别 + 任务分解 + Agent 路由；使用 LLM 做意图分类，预定义 Agent 能力矩阵做匹配；支持串行和并行 Agent 调用 |
| **"RAG 怎么避免幻觉？"** | Query Rewriting 提升检索精度；Two-Stage Retrieval (粗排+精排)；来源引用可追溯；Prompt 中明确要求"如果知识库中没有相关信息，请如实告知" |
| **"批改怎么保证公平？"** | Rubric 多维度评分（关键词覆盖度+语义相似度+逻辑完整性）；客观题精确匹配不受 LLM 影响；简答题可设置人工复核入口 |
| **"SM-2 算法怎么用的？"** | 不是简单的间隔重复；结合错题频次调整优先级；易混概念对检测与 SM-2 联动；每日任务生成自动按紧急度排序 |
| **"多模型怎么选的？"** | 按任务复杂度分层：简单生成用 DeepSeek（低成本），中等推理用 GPT-4.1-Mini（性价比），复杂批改/分析用 GPT-4.1 或 Claude（强推理）；模型抽象层支持热切换 |

### 10.3 技术关键词（简历标签）

```
Multi-Agent Orchestration ｜ Agentic Workflow ｜ RAG Pipeline
LLM-as-Judge ｜ Prompt Engineering ｜ Structured Output
Vector Embedding ｜ SM-2 Spaced Repetition
FastAPI ｜ ChromaDB ｜ LangChain ｜ DeepSeek API
```

### 10.4 项目可以延伸的方向（面试加分）

- **Agent 间自主对话**：Grading Agent 发现薄弱点 → 自动通知 Quiz Agent 出针对性题目
- **多模态学习**：支持图片（架构图 OCR）、音频（课程录音转文字）
- **协作学习**：多人学习小组，共享错题本和笔记
- **学习分析报告**：每周/每月自动生成学习分析报告（PDF）
- **Web 端 + 移动端适配**：PWA 支持离线学习
