### 2. HTTP 和 RESTful API 基础

**是什么：** 浏览器/前端怎么和后端"对话"的协议。

**通俗比喻：**
- 你去餐厅 → 你是 **前端**（客户端）
- 服务员 → 是 **API**（接口）
- 厨房 → 是 **后端**（服务器）
- 你点菜（**Request**）→ 服务员传话 → 厨房做菜 → 服务员端给你（**Response**）

**你需要理解的概念：**

```
GET     → 查询  （"给我看看笔记列表"）
POST    → 创建  （"帮我生成一篇新笔记"）
PUT     → 更新  （"修改这篇笔记的标题"）
DELETE  → 删除  （"删掉这篇笔记"）

URL = http://localhost:8000/api/v1/notes/generate
      ──────────┬────────── ─┬─ ───┬─── ─┬───
               主机地址       API版本  资源   动作
```

**在这个项目里：** 前端点"生成笔记"按钮 → 发 POST 请求到 `/notes/generate` → 后端干活 → 返回结果给前端

---

### 3. FastAPI 是什么

**是什么：** Python 的一个 Web 框架，帮你快速搭建 API 服务。

**为什么选它：**
- 自带异步支持（同时处理多个请求不卡顿）
- 自动生成 API 文档（访问 `/docs` 就能看到所有接口）
- 支持 SSE 流式输出（AI 生成内容时可以一个字一个字往外蹦，而不是等全部生成完再显示）

**最简示例让你感受一下：**

```python
from fastapi import FastAPI

app = FastAPI()

@app.post("/notes/generate")
async def generate_note(topic: str):
    # 这里调用 AI 生成笔记...
    result = await call_llm(f"请整理关于{topic}的笔记")
    return {"title": topic, "content": result}
```

**你需要学到什么程度：** 能写路由、能处理请求参数、能返回 JSON、能跑起来一个服务

## 第二层：AI/Large Language Model 核心知识

### 4. 什么是 LLM（大语言模型）

**是什么：** 就是 ChatGPT、DeepSeek 背后那个"大脑"。你输入文字，它输出文字。

**通俗理解：** 一个超级"文本接龙机器"——你给它一段话（Prompt），它根据海量训练数据，预测最合理的下文。

**你需要理解的核心概念：**

```
┌─────────────────────────────────────────────┐
│                    LLM                        │
│                                               │
│  输入 (Prompt)        输出 (Completion)        │
│  "翻译成英文：你好"  →  "Hello"                │
│  "1+1等于几？"       →  "2"                   │
│  "帮我写一首诗"      →  "春眠不觉晓..."        │
│                                               │
│  关键参数:                                     │
│  - Temperature (0-2): 越高越有创意，越低越严谨 │
│  - Max Tokens: 最多输出多长                    │
│  - System Prompt: 给模型设定"人设"             │
└─────────────────────────────────────────────┘
```

**Token 是什么：** LLM 不认字，只认 Token。大致 1 个中文字 ≈ 1-2 Token，1 个英文单词 ≈ 1-2 Token。API 按 Token 数量收费。

**在这个项目里：**
- DeepSeek：便宜，用来生成笔记、出题
- GPT-4.1 / Claude：能力强，用来批改简答题、分析薄弱点
- Ollama：跑在本地的免费模型，当你没网或不想花钱时用

### 5. Prompt Engineering（提示词工程）

**是什么：** 学会"怎么问 AI 才能得到好答案"。这是这个项目的核心技能。

**为什么重要：** 同一个模型，Prompt 写得好坏，结果天差地别。

**三种核心技巧你项目里都会用到：**

#### ① System Prompt（设定人设）

```
❌ 烂: "出几道题"
✅ 好: "你是一位拥有 10 年经验的 ISTQB 认证软件测试讲师。
       请根据以下知识点生成 5 道单选题，要求：
       - 难度：中等
       - 每题 4 个选项
       - 只有一个正确答案
       - 附上解释"
```

#### ② Few-Shot Prompting（给例子）

```
❌ 烂: "把这段文字整理成笔记"
✅ 好: "请把以下内容整理成结构化笔记。
       参考格式：
       ## 标题
       ### 子标题
       - 要点1：解释
       - 要点2：解释
       
       示例输入 → 示例输出：[给一个完整的例子]
       
       现在请处理：[实际内容]"
```

#### ③ Prompt Chaining（链式调用）

```
不要一次性让 AI "提取知识 + 整理结构 + 生成笔记"
而是拆成 3 步：
  第1步 Prompt："从这段文字中提取所有关键概念"
  第2步 Prompt："将以下概念按层级分类组织"
  第3步 Prompt："将结构化知识转化为 Markdown 笔记"
  
每步的输出作为下一步的输入 → 质量远超一步到位
```

### 6. Structured Output（结构化输出）

**是什么：** 让 AI 返回的不是一段自由文本，而是严格格式的 JSON。

**为什么要：** 因为前端/代码需要解析 AI 的输出。如果 AI 返回格式不稳定，你的程序就崩了。

**示例：**

```python
# 你在代码中要求 AI 必须按这个 JSON Schema 返回
schema = {
    "type": "object",
    "properties": {
        "title": {"type": "string"},
        "sections": {
            "type": "array",
            "items": {
                "type": "object",
                "properties": {
                    "heading": {"type": "string"},
                    "content": {"type": "string"}
                }
            }
        }
    },
    "required": ["title", "sections"]  # 这两个字段必须要有
}

# AI 就只会返回这样的 JSON，不会多也不会少：
# {"title": "...", "sections": [{"heading": "...", "content": "..."}]}
```

## 第三层：Agent 架构（本项目的核心）

### 7. 什么是 AI Agent

**是什么：** Agent = LLM + 工具 + 自主决策。

```
普通 LLM 用法：
  你问 → 它答 → 结束

Agent 用法：
  你给目标 → Agent 自己规划步骤 → 调用工具 → 检查结果 → 不够好就再来
```

**通俗比喻：**
- LLM 是"大脑"（会思考）
- Agent 是"人"（有大脑 + 有手去拿工具 + 有目标去执行）
- Tool（工具）= 函数调用（查数据库、搜文件、调 API）

**在这个项目里，每个 Agent 都是一个"专业角色"：**

```
Note Agent    = "我是一个笔记专家，我擅长整理知识"
Quiz Agent    = "我是一个出题老师，我擅长出各种类型的题目"
Grading Agent = "我是一个批改老师，我擅长评分和给出建议"
Memory Agent  = "我是一个记忆管理员，我记录你哪里容易出错"
```

---

### 8. Multi-Agent 协作

**是什么：** 多个 Agent 分工合作，像一个小团队。

**为什么不用一个 Agent 干所有事：**
- 单个 Prompt 太长会导致模型"分心"
- 不同任务需要不同的 System Prompt 人设
- 每个 Agent 的策略可以独立优化
- 修改一个 Agent 不影响其他

**Orchestrator（编排器）是"队长"：**

```
用户说："帮我出几道黑盒测试的题，做完给我批改"

         Orchestrator 收到后分析：
              │
              ├── 第1步：调用 Quiz Agent 出题（这个任务适合 Quiz Agent）
              │         等用户做完...
              │
              └── 第2步：把用户答案+题目 → 调用 Grading Agent 批改
```

---

### 9. Tool Calling（工具调用/Function Calling）

**是什么：** Agent 不只是"想"，还能"做"——调用你预先写好的函数。

**示例：Agent 需要查数据库，它不是自己"猜"，而是调用你的工具函数：**

```python
# 你定义的"工具"
def search_knowledge_base(query: str) -> list:
    """在知识库中搜索相关内容"""
    return vector_db.search(query)

# Agent 决定调用这个工具时：
# 1. 知道要调用 search_knowledge_base
# 2. 自动填入参数 query="黑盒测试方法"
# 3. 拿到返回结果
# 4. 基于结果生成最终回答
```

## 第四层：数据存储

### 10. Vector Database（向量数据库）+ Embedding

**这是 RAG 的基础，也是面试最爱问的。**

#### 先理解 Embedding（嵌入）

把一段文字变成一串数字（向量），语义相近的文字，数字也相近。

```
"苹果好吃"      → [0.1, 0.8, -0.3, 0.5, ...]  (1536个数字)
"这个水果很甜"   → [0.2, 0.7, -0.2, 0.6, ...]  ← 数字很接近！
"今天下雨了"    → [-0.5, -0.1, 0.9, -0.3, ...] ← 数字差很远！
```

#### 再理解 VectorDB

- 传统数据库：精确匹配查询 `WHERE name = '软件测试'`
- 向量数据库：语义搜索 "和这段话意思相近的内容"

```
用户问："怎么验证软件质量？"

传统搜索：找包含"验证"、"软件"、"质量"这些词的文档
向量搜索：找意思相近的内容 → 即使文章里用的是 "Verification"、"质量保证"
          也能找到！因为 Embedding 向量距离近。
```

#### 这个项目里 ChromaDB / FAISS 的作用

```
1. 你的所有笔记 → 切成小块 → 每块转成 Embedding → 存进 VectorDB
2. 用户提问 → 问题转成 Embedding → 在 VectorDB 中找最相似的 Top-5 块
3. 把找到的5块 + 用户问题 → 一起喂给 LLM → LLM 基于真实笔记回答问题
       ↑
    这就是 RAG（检索增强生成）——AI 的回答是"有据可查"的，不是瞎编的
```

### 11. SQL 关系型数据库

**是什么：** 用表格来存结构化数据。

**为什么这个项目需要 SQL 而不仅仅是 VectorDB：**
- VectorDB 擅长"语义搜索"但不擅长"统计聚合"
- SQL 擅长"这个用户错了几次？"、"错误最多的知识点是哪个？"

```
两个数据库分工：
  VectorDB: "找和 'Integration Testing' 语义相近的笔记"
  SQL:      "统计用户过去30天每个知识点的错误次数，排个名"
```

**你需要学的基本 SQL：**

```sql
SELECT * FROM notes WHERE user_id = 'u_001';           -- 查
INSERT INTO error_log (...) VALUES (...);               -- 增
UPDATE sm2_states SET ef = 2.6 WHERE id = 'sm2_001';   -- 改
DELETE FROM error_log WHERE id = 'err_001';             -- 删
```

## 第五层：进阶加分知识

### 12. SM-2 间隔重复算法

**是什么：** Anki（知名记忆软件）背后的算法。原理很简单：

```
你记一个知识点 → 刚学完马上复习（间隔1天）
              → 如果记住了 → 下次3天后复习
              → 又记住了   → 下次7天后复习
              → 又记住了   → 下次21天后复习
              ...
              → 忘了       → 重新从1天开始

EF (Easiness Factor)：你越容易记住，间隔增长越快
评分 0-5：你自评有多容易想起来
```

**在这个项目里：** 不是简单记录错题，而是计算每道错题"什么时候该重做"，然后每天给你推送最该复习的内容。

---

### 13. Reranking（重排序）

**是什么：** 向量搜索返回了 50 条结果，但不够精准 → 用更强的模型再筛一遍。

```
VectorDB 粗排（快但粗糙，Bi-Encoder）
  50条候选
     ↓
Reranker 精排（慢但精细，Cross-Encoder）
  挑出最相关的 5 条
     ↓
喂给 LLM 生成答案
```

## ❓ 常见疑问

**Q: LangChain 要不要学？**

A: 先了解它的概念（Chain、Agent、Tool 是什么），但不用深入。这个项目的核心是自己写 Agent 编排，学会原理比学会用框架更重要。LangChain 可以当工具库来用（比如它的 Document Loader、Text Splitter），但 Agent 编排逻辑自己写。

**Q: 前端 Streamlit 还是 Next.js？**

A: 先 Streamlit。它是纯 Python 写的，不需要学 JavaScript，10 分钟就能出一个能用的界面。Next.js 等 Phase 4 再学，那是为了面试展示效果。

**Q: 为什么有的模型选 DeepSeek，有的选 GPT？**

A: 省钱 + 分任务。生成笔记、出题这类简单任务，DeepSeek 够用且便宜几十倍。批改简答题、分析弱点这类需要强推理的复杂任务，才用 GPT-4.1 或 Claude。这是一个工程上的"性价比路由"设计——面试可以讲。

**Q: 我现在是纯小白，真能做出这个项目吗？**

A: 能。这个项目的每个模块拆开后都不复杂。先跟着学习路线走 2 周把 Python + FastAPI 基础打牢，后面就是"搭积木"。第一步的目标不是做完整项目，而是写出一行 `python main.py` 能调用 DeepSeek 返回一段文字。那一步跑通了，后面都是在这个基础上加东西。
